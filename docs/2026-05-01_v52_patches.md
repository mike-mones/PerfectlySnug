## Patch 1 — Override floor REMOVED

**User intent:** "if the user adjusts, that's not an 'override' it's a
learning data point." Manual changes feed the learner cross-night, but
they should NOT clamp the algorithm for the rest of the night.

### What changed

| Location | Before | After |
|---|---|---|
| `_state` init | had `override_floor: None` and `override_floor_blower_pct: None` slots | removed both slots |
| `_compute_setting` post-call (formerly lines 512-517) | clamped `setting` upward to stored floor | block deleted; brief comment explaining the change |
| `_log_to_postgres` call (line ~569) | passed `override_floor=floor` | passes `override_floor=None` |
| `_log_to_postgres` signature & notes formatting | unchanged | preserved for back-compat (`if override_floor is not None` guard handles None correctly) |
| `_on_sleep_mode("on")` reset block | reset both floor slots to None | floor slots removed |
| `_on_sleep_mode("off")` reset block | same | same |
| `_on_setting_change` (override handler) | computed `floor = min(new_val, MAX_SETTING)`, stored to state | floor computation removed; only `override_freeze_until`, `last_setting`, `override_count` are touched |

### What was preserved

- **60-min `override_freeze_until`** still applies on every override. The
  controller does not change L1 within 60 min of a manual change. After
  60 min, algorithmic decisions resume.
- **Kill switch** (3 manual changes in 5 min → manual-mode-for-night)
  still works.
- **Cross-night learner** (`_learn_from_history`) is unchanged — it queries
  PG `controller_readings WHERE action='override'` and was always the
  correct learning path; the floor was a separate within-night clamp.
- **`_log_to_postgres` `override_floor` kwarg** is preserved (always None
  now) for forward/backward compatibility.

### Why this addresses the user's request

v5.2's behavior was: user manually nudges to -5 → freezes for 60 min → after
freeze, controller algorithm wants -10 → **clamped to -5 for the rest of
the night.** The user said "the model is supposed to learn." Now: same
nudge → 60-min freeze → algorithm resumes, can pick anything in [-10, 0]
based on cycle/body/room. Next night, the override is one more data point
for `_learn_from_history`.

### Tests

`tests/test_controller_v5.py::TestOverrideFloor::test_override_floor_persists_after_freeze`
was rewritten to
`test_no_floor_after_freeze_returns_to_algorithmic` — it now asserts the
controller writes -10 (algorithmic value), not -5 (former floor), and that
the `override_floor` state slot is no longer present.

### Risks / things to monitor

- **User briefly feels "the controller is fighting me" after a manual
  change.** The 60-min freeze should cushion this, but if the algorithm
  walks back warmer/cooler immediately after the freeze, the user may
  override again. That itself is a learning signal.
- **Asymmetric override-floor problem (audit backlog H4)** is partially
  resolved: there is now NO floor for any override (warm OR cold).
  Previously cold overrides had no floor while warm did.

## Patch 2 — 3-level mode watchdog

**User intent:** "turning off 3 level schedule." User runs single-stage
L1; if 3-level mode drifts on, firmware advances L1→L2→L3 by run_progress
and writes to `bedtime_temperature` (L1) stop affecting the live dial.

### What changed

| Location | Change |
|---|---|
| Constants section | added `E_PROFILE_3LEVEL_RIGHT = "switch.smart_topper_right_side_3_level_mode"` |
| Method addition (next to `_ensure_responsive_cooling_off`) | new `_ensure_3_level_off()` method that turns OFF 3-level on BOTH sides; idempotent |
| `initialize()` | calls `_ensure_3_level_off()` once at boot |
| `_control_loop()` | calls `_ensure_3_level_off()` every 5-min tick (right after `_ensure_responsive_cooling_off()`) |
| `_on_sleep_mode("on")` | calls `_ensure_3_level_off()` at sleep start |

The pattern mirrors the existing `_ensure_responsive_cooling_off`
watchdog, which was already used the same way.

### Tests

No new tests. Existing tests pass without modification.

### Risks / things to monitor

- **Edge case: HA UI or external automation explicitly turns 3-level ON.**
  The watchdog will turn it back off within ≤5 minutes. If an automation
  is fighting us, that automation is the bug — file an issue.
- **If the topper firmware ever requires 3-level ON for some legitimate
  feature**, this watchdog would be wrong. Currently no such feature is
  used.

## Patch 3 — Right-zone proactive-cool rail at body ≥ 86°F sustained

**User intent:** wife's comfort. Synth fleet flagged the *override-absence
trap* (wife rarely overrides → not because comfortable, but because not
motivated). User accepted 84°F threshold; rubber-duck pass argued for
86°F because v5.2's existing right body feedback already corrects -2 at
body=84°F (Kp_hot=0.5 × delta=4 = 2 steps cooler from base). 86°F is the
next escalation, not redundant.

### What changed

New constants:

```python
RIGHT_HOT_RAIL_F = 86.0          # body_left threshold for the rail
RIGHT_HOT_RAIL_STREAK = 2        # 2 consecutive ticks (≈10 min @ 5-min cycles)
RIGHT_HOT_RAIL_BIAS = -1         # one step cooler when streak triggers
```

New state slot: `right_hot_streak` (int, persisted across AppDaemon
restarts via `_save_state` → `controller_state_v5_rc_off.json`).

New logic block in `_right_v52_shadow_tick` (right after `proposed` is
computed from base + body feedback + room comp):

```python
in_excluded_window = (
    in_initial_bed_cooling or in_bedjet_window or pre_sleep_stage
    or not occupied or body_skin is None
)
if in_excluded_window:
    self._state["right_hot_streak"] = 0
elif body_skin >= RIGHT_HOT_RAIL_F:
    self._state["right_hot_streak"] = self._state.get("right_hot_streak", 0) + 1
else:
    self._state["right_hot_streak"] = 0

right_hot_streak = self._state.get("right_hot_streak", 0)
hot_rail_fired = (right_hot_streak >= RIGHT_HOT_RAIL_STREAK
                  and not in_excluded_window)
if hot_rail_fired:
    pre_rail_proposed = proposed
    proposed = max(-10, proposed + RIGHT_HOT_RAIL_BIAS)  # -1 step cooler
    source += f"+hot_rail_{right_hot_streak}"
    self.log(f"Right hot-rail fired: body_left={body_skin:.1f}°F "
             f"streak={right_hot_streak} → {pre_rail_proposed:+d} → {proposed:+d}")
```

Streak resets in `_on_sleep_mode("on")` reset block (alongside
`hot_streak`).

### Why these specific design choices

- **Threshold 86°F not 84°F** — at 84°F the existing body_fb already gives
  the right zone -2 from base. Adding -1 more would compound. 86°F is a
  meaningful escalation; only 12 ticks (≈60 minutes) in the historical
  data crossed it on the right side.
- **Streak ≥2 (≈10 min)** — avoids triggering on transient spikes from
  blanket movement or BedJet residual. Tunable.
- **Excluded windows** — BedJet (sensors are inflated by warm air),
  initial-bed-cooling (already at -10), pre-sleep stages (intentional),
  unoccupied (no signal), sensor missing.
- **Bias -1 not -2** — incremental; multiple consecutive sustained-hot
  windows will naturally compound (each fresh streak start fires once).
  Hard cap at -10.
- **Acts on `body_skin = body_left`** — same channel as
  `right_overheat_safety.py` and consistent with the per-zone-calibrated
  finding that the right zone's body_left p99 ≈ 88°F (left zone p99
  also ≈88; both runs warm but right has wider tail).

### Coordination with `right_overheat_safety.py`

`right_overheat_safety.py` engages at body_left ≥ 86°F sustained 2 ticks
and forces L1=-10. The right_hot_rail in this controller fires at the
**same threshold**. This is intentional: the rail forces -10 unconditionally,
while the controller's hot_rail is a softer -1-step bias that fires before
or at the same threshold. In practice:

1. Body crosses 86°F. Both apps notice on the same tick or adjacent ticks.
2. Controller's hot_rail biases proposed setting one step cooler.
3. If body stays ≥86°F (streak builds → also ≥2 in `right_overheat_safety`),
   right_overheat_safety force-writes L1=-10 directly, overriding the
   controller's softer write.
4. Both write to the same entity; last-writer-wins. There is currently
   NO mutex (red-safety §1 flagged this). In practice it's fine because
   the rail's force-write is monotonically more aggressive; the
   controller's softer write would not undo it.

**This is a known race** (red-safety §1). Tonight's deploy does not fix
it. If the rail engages and the controller writes a warmer value on the
following tick, we could see one cycle of bouncing. Monitor in morning
report.

### Tests

No new dedicated tests for the rail logic in tonight's commit. Test
coverage is in the existing `test_right_overheat_safety.py` (rail
behavior) and `test_controller_v5.py` (controller behavior). A dedicated
`test_right_hot_rail_*` would be a good follow-up.

### Risks / things to monitor

- **Wife's body_left rarely crosses 86°F** historically. If the rail never
  fires, no behavior change vs v5.2. If it fires multiple times per night,
  that's actually the system catching previously-silent overheats.
- **Streak reset on missing sensor** could be flaky if `body_left` reads
  briefly None during a normal cycle. Streak resets prevent runaway, but
  could under-fire. Tradeoff accepted.
- **Cooldown after firing**: there is none. If body stays ≥86°F for 20
  ticks, the rail fires every tick. The proposed setting is biased -1
  each time, but rate-limit on the actuation side
  (`RIGHT_MIN_CHANGE_INTERVAL_SEC=1800`) prevents rapid re-writes.

## Patch 4 — BedJet → -10 force on right zone: NOT a code change

User said: "bedjet is just the first 30 minutes of sleep blasting heat...
it should mean that the bedjet goes to -10 for the time the bedjet is
blasting heat."

**v5.2 already does this.** The `INITIAL_BED_RIGHT_SETTING = -10` constant
forces the right side to -10 for the first `INITIAL_BED_COOLING_MIN = 30.0`
minutes after right-bed onset (verified at `_right_v52_shadow_tick` line
~715). The BedJet window is also 30 min (`RIGHT_BEDJET_WINDOW_MIN = 30.0`,
matched to `right_overheat_safety.BEDJET_SUPPRESS_MIN`). So v5.2 already
pins right=-10 throughout the BedJet warm-blanket period.

The only thing v5.2 does NOT yet do:

- **Exclude BedJet-window data from preference learning.** User said BedJet
  data "shouldn't be taken into account when determining preferences."
  `_learn_from_history` queries PG `action='override'` events. If she
  manually overrides during the BedJet window, that override IS counted.
  No fix tonight; queued in `docs/NEXT_STEPS.md`.
