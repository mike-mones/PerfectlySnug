## 1. Patch sequence (P1 → P11, strictly ordered)

### Conventions

- **Phase** maps to architecture doc §4: A=ingestion, B=shadow state,
  C=shadow policy, D=cutover, E=learner.
- **Mode** is **SHADOW** (logs only, v5.2 still actuates) or **LIVE** (this
  patch's code controls the dial). Never both.
- LOC counts are estimates. If a patch exceeds ~400 LOC, split it (and
  re-number the rest).
- Every patch ships behind a feature flag (HA helper or Python const) that
  defaults OFF; the patch's "Behavior change" section names the flag.
- Every PG migration in this plan is **additive only** (`ALTER TABLE …
  ADD COLUMN IF NOT EXISTS …`). No `DROP`, no rewrites. Symmetric
  `_rollback.sql` ships in the same patch.
- Tests must pass *before* SCP. AppDaemon will auto-reload Python files on
  change; no addon restart is required for `appdaemon/*.py` edits unless
  the patch says otherwise.

## 2. Promotion gates between patches

After **every** patch:

1. **Required observation period:**
   - SHADOW patches (P1, P2, P3, P4, P5, P8): **≥1 occupied night** of logging.
   - LIVE patches (P6, P7, P9, P10, P11): **≥3 paired nights** per `evaluation.md §5`.
   - HOLD path: extend by N nights (decision logged in CHANGELOG); do **not** advance to the next patch.
2. **Required morning subjective report** (`input_number.snug_morning_comfort_rating`, 1–5): logged via HA mobile shortcut. Per `evaluation.md §8`, until Spearman ρ between `adj_weighted_score` and the rating is established (≥14 reports, ρ ≥ 0.5), the §5 ACCEPT thresholds are *advisory* and changes additionally require a user "yes."
3. **Required `eval_compare.py` run:**
   ```bash
   .venv/bin/python tools/eval_compare.py \
       --A-version "<previous-version-tag>" \
       --B-version "<this-patch-version-tag>" \
       --zone <left|right>
   ```
   Print result. Exit code: `0` ACCEPT / `1` HOLD / `2` REVERT. Commit the decision (single line in CHANGELOG.md: `2026-05-XX P<n> ACCEPT|HOLD|REVERT — <one-line rationale>`).
4. **Decision matrix (per `evaluation.md §5`):**
   - **ACCEPT:** advance to next patch.
   - **HOLD:** explicit `--extend-observation +N` annotation in CHANGELOG; re-run after N nights. Never silently linger.
   - **REVERT:** run the patch's REVERT command verbatim. Re-open the patch with a follow-up issue.

## 3. Anti-goals (explicit; refuse-class)

These are **non-negotiable**. A reviewer who sees one of these in a proposed patch must reject it.

- **NO** big-bang cutover. P6 (left) and P7 (right) MUST be separate patches with ≥7 nights of left-live observation between them.
- **NO** re-introduction of `cycle_num`, `CYCLE_SETTINGS`, `RIGHT_CYCLE_SETTINGS`, `_get_cycle_num`, `_normal_cool_base`, `_cycle_index`, `RegimeConfig.cycle_baseline_left/right`, `night_progress`, `sin_cycle`, `cos_cycle`, or any time-of-night-keyed primary lookup. (`seconds_since_presence_change` as a *weak prior* on `WAKE_TRANSITION` per `state_estimation.md §5.1` is the only allowed time-axis use; a tiebreaker with confidence cap 0.5 in degraded mode per `state_estimation.md §5.2` is the second.)
- **NO** deep learning, neural networks, LSTM, transformer, or any model >100 parameters. The learner is **16 floats**, period (per `learning.md §0`).
- **NO** offline retraining pipeline. The learner is online and incremental. Population prior is regenerated quarterly via a CLI (`learning.md §6.4`).
- **NO** removing `right_overheat_safety.py`, the `body_fb` occupancy gate, the body-sensor validity gate, the override freeze, or the two-key arming pattern. Any proposal "for cleanliness" is a regression.
- **NO** `ALTER TABLE` that drops or rewrites existing columns. Additive only. Symmetric `_rollback.sql` ships in the same commit (per `evaluation.md §2` mandate).
- **NO** patch that touches more than one architecture layer. If a change requires touching L1 and L2 simultaneously, split it.
- **NO** patch that touches more than one zone simultaneously (P6 and P7 enforce this for cutover).
- **NO** new N-agent fleet design pass. One trusted authoring loop per change.
- **NO** waiting for "150+ overrides" or any sample-size gate. The system ships with `MIN_EVENTS_TO_UPDATE = 1` and a population prior (`learning.md §5.2`).
- **NO** new framework, new language, new message bus, new web UI. AppDaemon + Python + HA helpers + Postgres remain the entire stack (architecture §6).
