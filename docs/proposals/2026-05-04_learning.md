**Author:** learning-layer pass
**Status:** design proposal — replaces `ml/learner.py` and
`sleep_controller_v5.py::_learn_from_history`
**Audit reference:** `docs/findings/2026-05-04_context_audit.md` finding #3

**Companion docs:** `docs/proposals/2026-05-01_recommendation.md` §6 (policy

---

## 0. Goals & non-goals

**Goal.** Cross-night, per-(user, latent_state) **bias offset** that survives
the audit's three failure modes:

1. Override delta is treated as point-in-time ground truth → use a
   **temporal credit kernel**, not the instantaneous label.
2. "No overrides → no adjustments" → ship a **non-zero population prior**
   on day 1.
3. Contaminated label corpus (initial-bed, BedJet, pre-sleep, body-sensor
   empty-bed reads) → reuse the v5.2 SQL filters and add an
   `is_valid_for_learning` gate computed in Python.

**Non-goals.** No deep learning. No nightly batch retrain pipeline. No
Bayesian Ridge / LCB / GP / sklearn dependency (audit §5 deferred
`residual_head.py`). No multi-night batch fits — every event is consumed
incrementally. No new policy regime; offsets layer on top of `ml/v6/policy.py`.

The learned object is intentionally tiny: **one float per (user, state),
plus one per-user global, bounded in `[−3, +3]`**. Inspectable as a
single JSON file.

## 1. State taxonomy used by the learner

The learner operates on a **fixed taxonomy of 7 latent states**, derived
from the v6 regime classifier (`ml/v6/regime.py`) plus a discomfort
overlay. We deliberately do **not** invent a new set; we collapse v6
regimes into the smallest set that admits per-state preference learning.

| Latent state            | Source mapping                                                                                       |
|-------------------------|------------------------------------------------------------------------------------------------------|
| `PRE_SLEEP`             | `regime ∈ {PRE_BED}` OR `is_sleeping=False AND bed_occupied=True`                                    |
| `INITIAL_COOL`          | `regime == INITIAL_COOL` (first ≤30 min of bed onset, or ≤45 if room <66°F)                          |
| `STABLE_SLEEP_DEEP`     | `regime == NORMAL_COOL` AND elapsed_min ∈ [30, 240] AND `body_skin_f ∈ [78, 84]` AND no movement spike|
| `STABLE_SLEEP_REM`      | `regime == NORMAL_COOL` AND elapsed_min > 240 AND sleep_stage ∈ {`rem`, `core`}                      |
| `RESTLESS`              | `movement_density_15m > 0.10` (≥2× p75 baseline per `ml/v6/policy.py:259`) OR `body_30m_sd > p90`    |
| `WAKE_TRANSITION`       | `regime == WAKE_COOL` OR sleep_stage ∈ {`awake`, `wake`}                                             |
| `HOT_FLASH`             | `body_hot_f > 84.5` AND `body_trend_15m > +0.40 °F/15m`                                              |

**State exclusions (no offset learned, no offset applied):**
`UNOCCUPIED`, `OVERRIDE` (freeze active), `SAFETY_YIELD` (right rail),
`BEDJET_WARM` — these are governed by deterministic rules and their
"correct" setting is policy-dictated, not preference-dictated.

Off-bed and pre-sleep are also **excluded from the credit window** even
when the adjustment itself happened later — see §2.4.

User dimension is `{primary, partner}` keyed by zone (`left=primary`,
`right=partner`); cross-zone is never blended.

## 2. Feedback signal — the `AdjustmentEvent`

### 2.1 Definition

```python
# ml/learning/event.py (new module)
from dataclasses import dataclass, field
from datetime import datetime
from typing import Optional, Literal

LatentState = Literal[
    "PRE_SLEEP", "INITIAL_COOL", "STABLE_SLEEP_DEEP", "STABLE_SLEEP_REM",
    "RESTLESS", "WAKE_TRANSITION", "HOT_FLASH",
    # excluded — present so the dataclass round-trips, but never learned on:
    "UNOCCUPIED", "OVERRIDE_FREEZE", "SAFETY_YIELD", "BEDJET_WARM", "PRE_BED",
]

@dataclass(frozen=True)
class TrajectoryPoint:
    t_offset_sec: int          # negative seconds before t_adjust (≤ 0)
    state: LatentState
    body_avg_f: Optional[float]
    body_skin_f: Optional[float]
    body_hot_f: Optional[float]
    room_f: Optional[float]
    movement_rms: Optional[float]
    setting: int               # the dial value at this moment
    setpoint_f: Optional[float]

@dataclass(frozen=True)
class AdjustmentEvent:
    # Identity
    event_id: str              # UUID; used for idempotent PG INSERT ON CONFLICT
    t_adjust: datetime         # UTC; truncated to seconds
    night_date: str            # America/New_York "night of" key, YYYY-MM-DD
    zone: Literal["left", "right"]
    user: Literal["primary", "partner"]

    # Action
    pre_setting: int           # what the controller had written
    post_setting: int          # what the user dialed to
    delta: int                 # post - pre  (negative = "too hot, cool me")

    # State at t_adjust
    state_at_adjust: LatentState
    sleep_stage_at_adjust: Optional[str]
    elapsed_min_at_adjust: float

    # Trajectory: 30 min trailing window, polled at the controller cadence
    # (5 min) plus the high-rate snapshot at t_adjust. Empty trail (cold
    # start, AppDaemon restart) is allowed; learner falls back to point
    # credit on state_at_adjust.
    trajectory: tuple[TrajectoryPoint, ...] = field(default_factory=tuple)

    # Source distinction (best-effort, may be None)
    # 'app' = HomeAssistant input_number / native control
    # 'physical' = topper button press (no HA event source attribution)
    # 'voice' = Assist / Siri Shortcut
    source: Optional[Literal["app", "physical", "voice"]] = None

    # Capture-time provenance
    controller_version: str = ""
    controller_patch_level: str = ""
```

### 2.2 Capture site

Wire **inside** `_on_setting_change` and `_on_right_setting_change` in
`appdaemon/sleep_controller_v5.py` (currently
:1259-:1306 and :1309-:1380). Two lines after the existing
`self._log_override(...)` call, add:

```python
event = self._build_adjustment_event("left", old_val, new_val)
if event is not None:
    self._learning.observe(event)         # see §4
```

`_build_adjustment_event` reads the in-memory rolling trail
(`self._state["snapshot_trail_left"]`), filters per §2.4, and returns
either the event or `None`.

The rolling trail is maintained by `_log_to_postgres`
(`sleep_controller_v5.py:2390+`) at every 5-min cycle: append the
snapshot dict to `self._state["snapshot_trail_<zone>"]`, capped at 7
entries (≈30 min of cycles) plus pruning by `t > now − 1800s`.

### 2.3 HA event source attribution

`_on_setting_change` receives `kwargs` from AppDaemon's `listen_state`
callback. The `context.user_id` field on the event (available via
`self.get_state(entity_id, attribute='context')`) distinguishes:

- **`app`** — `context.user_id` resolves to `person.mike` or
  `person.partner` via `auth.list_users()` mapping.
- **`physical`** — `context.user_id is None` AND old→new transition
  did **not** originate from a controller `_set_l1` call (we already
  track this via `expected = self._state["last_setting"]`).
- **`voice`** — `context.parent_id` is set and the parent is an Assist
  pipeline event.

This is best-effort. `source = None` is acceptable; the learner does
not condition on source for the offset update (it is informational and
used by §6 oscillation diagnostics only).

### 2.4 Capture filters (label hygiene)

An event is **emitted** to the learner only if all hold:

1. `t_adjust − last_adjust_t > 60 s` (debounce: a fix-fix is recorded
   to PG for replay but not to the learner).
2. `state_at_adjust ∉ {UNOCCUPIED, BEDJET_WARM, PRE_BED}`.
3. `mins_since_onset >= 10` — body sensor invalid window per audit
   §2.1. Specifically: with `body_skin_f − room_f < 6 °F` we treat
   the read as mattress equilibrium (per `RIGHT_BODY_SENSOR_VALID_DELTA_F`,
   `sleep_controller_v5.py:176-177`) and **drop the event**, not just
   downweight it. Empty-bed teaches us nothing.
4. `pre_setting != post_setting` (a no-op write from the controller's
   own `set_value` re-issue is not an adjustment).
5. `|delta| <= 10` (sanity; clamps any HA bus glitch).
6. The trajectory must contain ≥ 1 point with `t_offset_sec ≥ -1800`.
   If not (cold start), record event with empty trajectory and
   `state_at_adjust` only — learner uses point credit, see §3.4.

Filter (3) is the **fail-closed body-sensor patch generalized to
learning**, not just to control. Today's controller honors it on
actuation (`bodyFbFailClosed` patch, audit §2.1) but the learner still
ingests the bad row.

### 2.5 PG persistence

Events are also written to a new table `learning_events` for replay
and audit. The learner's in-memory update happens on observe; PG is
the durable record.

```sql

```

## 3. Credit assignment kernel

This is the audit's central request. Discomfort builds; the user
overrides at the point they decide to act, not at the point the
controller was first wrong.

### 3.1 Kernel

Define an exponential temporal weighting on the trailing window:

> **`w(τ) = exp(−λ · τ)`** for `τ ∈ [0, T_max]`, else `0`
>
> with **`λ = ln(2) / 10 min ≈ 0.0693 / min`** (10-minute half-life)
> and **`T_max = 30 min`** (truncation; w(30) ≈ 0.125).

`τ` is **minutes before adjustment**, i.e. `τ = (t_adjust − t_i) / 60s`.

Truncating at 30 min keeps blame inside one half-cycle; longer windows
start crediting prior states (e.g. INITIAL_COOL → STABLE_SLEEP) and
contaminate. The 10-min half-life matches the user's reported
"5–15 minute build-up" timescale (audit §3.2).

### 3.2 Credit per state

Given trajectory points `{(t_i, state_i)}_{i=1..N}` plus the boundary
point at `t_adjust` itself (state = `state_at_adjust`):

```
raw_w_i = exp(-λ · max(0, (t_adjust - t_i)/60))     for τ_i ∈ [0, 30]
                  0                                  otherwise

W_total = Σ_i raw_w_i
credit[s] = (Σ_{i: state_i = s} raw_w_i) / W_total       — normalized to sum 1
```

States in the exclusion set (§1) contribute 0 to numerator but still
to denominator (so blame can't "leak" to the wrong place: if the user
spent 80% of the trail in `BEDJET_WARM`, only 20% of credit goes to
the learnable states).

### 3.3 Worked example

User in bed 90 min, room 71 °F. Trajectory (one point per 5 min):

| τ (min) | state               | raw w   |
|---------|---------------------|---------|
|  0      | STABLE_SLEEP_DEEP   | 1.000   |
|  5      | STABLE_SLEEP_DEEP   | 0.707   |
| 10      | RESTLESS            | 0.500   |
| 15      | RESTLESS            | 0.354   |
| 20      | STABLE_SLEEP_DEEP   | 0.250   |
| 25      | STABLE_SLEEP_DEEP   | 0.177   |
| 30      | INITIAL_COOL        | 0.000   |  ← truncated

`W_total = 2.988`
`credit[STABLE_SLEEP_DEEP] = (1.000 + 0.707 + 0.250 + 0.177) / 2.988 = 0.714`
`credit[RESTLESS] = (0.500 + 0.354) / 2.988 = 0.286`
`credit[INITIAL_COOL] = 0.000` (truncated)

If the user dialed `−5 → −8` (delta = −3, "too hot"), the per-state
update before caps (η=0.20 — see §4) is:

```
Δoffset[STABLE_SLEEP_DEEP] = 0.20 · sign(-3) · 0.714 = -0.143
Δoffset[RESTLESS]          = 0.20 · sign(-3) · 0.286 = -0.057
```

(Note: we use `sign(delta)` not `delta`. Magnitude is captured by the
event count, not by the size of any single twist of the dial. Larger
adjustments are not proportionally more reliable — they often
overshoot. Keeping per-event step ≤ 0.2 prevents one panic-twist from
moving the bias by more than one notch.)

### 3.4 Cold-start fallback (no trajectory)

If `len(trajectory) == 0` (AppDaemon restart, first-night):
`credit[state_at_adjust] = 1.0`, all other states 0. Update rule
proceeds normally.

## 4. Learned function shape

### 4.1 Parameters

```python
# Bounded per-(user, state) bias offset (units: dial steps)
offset[user, state] : float ∈ [-3.0, +3.0]

# Optional per-user global bias (the "this person runs warm" knob)
offset_global[user] : float ∈ [-2.0, +2.0]
```

Total parameter count:
`2 users × 7 learnable states + 2 globals = 16 floats.`

Stored in a single JSON file (§7).

### 4.2 How offsets enter the policy

In `ml/v6/policy.py::compute_v6_plan`, immediately after the existing
`combined = base_setting + body_fb + room_fb + proxy_term` line
(currently :155):

```python
# §4.2 of 2026-05-04_learning.md — apply learned per-state offset
# AFTER deterministic terms, BEFORE residual head, BEFORE clamp.
learned_offset = learning_store.get_effective_offset(
    user=zone_to_user(zone),
    state=regime_label_to_latent_state(regime_label, snapshot),
)
combined += learned_offset
```

`learned_offset = offset[user, state] + offset_global[user]`, rounded
to nearest int **only at clamp time** (so multiple sub-step offsets
compound correctly — e.g. `−0.4 + (−0.3)` rounds to `−1`, not to `0`).

The **sign convention** matches the policy: `combined` is the dial
value (negative = colder). A learned offset of `−1.0` means "this user
wants ~1 step colder than the deterministic policy in this state,"
which is exactly the polarity of the per-event update in §3.3.

Offsets are **never applied** when the regime is `OVERRIDE`,
`UNOCCUPIED`, `SAFETY_YIELD`, `INITIAL_COOL`, or `PRE_BED` — these
are policy-dictated, not preference-dictated. (Note that we still
*learn* from a `STABLE_SLEEP_DEEP` adjustment whose 30-min trail
included some `INITIAL_COOL` minutes — but the credit on
`INITIAL_COOL` is dropped per §1 exclusions; only the learnable
fraction remains.)

### 4.3 Why bounded, not unbounded regression

Audit §5 line 311 retired `residual_head.py`'s 448-LOC sklearn
BayesianRidge for exactly this reason: a free-form learned function
on n=53 noisy events is over-parameterized. A 16-float lookup with
hard clamps is:

- inspectable (the file is a JSON object you can `cat`),
- reversible (delete the file → behavior reverts to deterministic v6),
- bounded in damage (max ±5 dial steps total: ±3 state + ±2 global).

## 5. Update rule

### 5.1 Online incremental update

For each `AdjustmentEvent` that passes filters:

```python
def observe(event: AdjustmentEvent) -> dict:
    if event.state_at_adjust in NON_LEARNABLE: 
        return {"rejected": "non_learnable_state"}

    credits = compute_credit_weights(event.trajectory, event.t_adjust)
    sgn = math.copysign(1.0, event.delta)            # +1 or -1; never 0 (filter (4))
    user = event.user

    night_caps = _per_night_caps_remaining(user, event.night_date)

    deltas = {}
    for state, w in credits.items():
        if state in NON_LEARNABLE or w == 0.0:
            continue
        eta = effective_eta(user, state)             # see §5.2
        raw = eta * sgn * w
        # Per-night cap on this state for this user
        capped = clamp(raw, -night_caps[state], +night_caps[state])
        deltas[state] = capped
        offset[user][state] = clamp(
            offset[user][state] + capped, -3.0, +3.0
        )
        n_events[user][state] += 1
        update_oscillation_buffer(user, state, sgn)

    # Global bias: ALWAYS gets a fractional update from every event,
    # because "this user runs warm" is the residual that no per-state
    # offset can absorb. Smaller learning rate.
    eta_g = ETA_GLOBAL / (1 + n_events_global[user] / 20.0)
    raw_g = eta_g * sgn * 1.0
    capped_g = clamp(raw_g, -night_caps["__global__"], +night_caps["__global__"])
    offset_global[user] = clamp(
        offset_global[user] + capped_g, -2.0, +2.0
    )
    n_events_global[user] += 1

    return {"deltas": deltas, "global_delta": capped_g, "credits": credits}
```

### 5.2 Hyperparameters

| Symbol                   | Value                          | Source / rationale                                      |
|--------------------------|--------------------------------|---------------------------------------------------------|
| `ETA_BASE`               | `0.20`                         | one event ≈ 1/5 dial step; 5 consistent events ≈ 1 step |
| `effective_eta(u, s)`    | `ETA_BASE / (1 + n[u,s]/10)`   | decays with local sample count (sample-size shrinkage)  |
| `ETA_GLOBAL`             | `0.05`                         | global moves ¼ as fast as per-state                     |
| `LAMBDA` (kernel)        | `ln(2)/10 min`                 | §3.1 — 10-min half-life                                 |
| `T_MAX_KERNEL`           | `30 min`                       | §3.1 — truncation                                       |
| `OFFSET_HARD_CAP`        | `±3.0`                         | one full sleep cycle's worth of "I run warm/cold"       |
| `OFFSET_GLOBAL_CAP`      | `±2.0`                         | partial; user-level bias is rarer to need               |
| `PER_NIGHT_STATE_CAP`    | `±1.0`                         | so one bad night cannot blow up a state                 |
| `PER_NIGHT_GLOBAL_CAP`   | `±0.5`                         | global moves at most ½ step per night                   |
| `MIN_EVENTS_TO_UPDATE`   | `1`                            | **explicit reject of "needs more data" failure mode**   |
| `OSCILLATION_WINDOW`     | last 3 update signs            | §6.2                                                    |
| `HOLDOUT_NIGHTS`         | last 3 nights                  | §6.1                                                    |

`MIN_EVENTS_TO_UPDATE = 1` is the inverse of audit §2.8. The current
learner gates LightGBM at 150+ events; v6 gates the residual head at
14 nights; **this learner updates after the first event because the
prior (§5) ensures we are never starting from zero.**

### 5.3 Per-night caps

Caps reset at `_on_sleep_mode("on")`. Tracking is in-memory and
persisted to JSON alongside the offsets so they survive AppDaemon
restart mid-night. Cap remaining for `(user, state)` is
`max(0, PER_NIGHT_STATE_CAP − Σ |Δoffset| applied tonight)`. A new
event is permitted to apply the residual cap (so the 5th big event
of the night gets a tiny update, not zero — softer than a hard wall).

## 6. Day-1 prior (population offsets)

### 6.1 Strategy

Effective offset blends user history and population:

> **`effective_offset[u, s] = α(n[u,s]) · offset[u, s] + (1 − α(n[u,s])) · prior[s]`**
>
> with **`α(n) = 1 − exp(−n / 10)`** (≈0.10 at n=1, 0.39 at n=5, 0.86 at n=20).

`prior[s]` is the **same scalar for both users** at v1; computed
offline from the existing `controller_readings` corpus and committed
as a JSON file (`ml/learning/population_prior.json`, regenerated by
a CLI tool).

### 6.2 SQL to derive the prior

The prior is the **population mean of (override_setting − controller_setting)**
per latent state, computed from contamination-filtered override rows.

```sql
-- ml/learning/derive_population_prior.sql
WITH classified AS (
  SELECT
    cr.zone,
    cr.ts,
    cr.setting        AS override_setting,
    cr.effective      AS controller_setting,
    (cr.setting - cr.effective)::REAL AS delta,
    cr.elapsed_min,
    cr.regime,                                     -- v6 column, NULL for v5.x rows
    cr.sleep_stage,
    cr.body_left,
    cr.body_center,
    cr.ambient,
    cr.movement_density_15m,
    cr.bed_presence,
    cr.notes
  FROM controller_readings cr
  WHERE cr.action = 'override'
    AND cr.setting   IS NOT NULL
    AND cr.effective IS NOT NULL
    AND cr.setting IS DISTINCT FROM cr.effective
    AND cr.ts > now() - interval '180 days'
    AND COALESCE(cr.bed_presence, 'on') = 'on'
    -- Reuse the v5.2 contamination filters verbatim
    AND COALESCE(cr.notes, '') NOT LIKE '%initial_bed_cooling%'
    AND COALESCE(cr.notes, '') NOT LIKE '%bedjet_window%'
    AND COALESCE(cr.notes, '') NOT LIKE '%pre_sleep%'
    -- Body-sensor empty-bed gate (audit §2.1, generalized)
    AND COALESCE(cr.body_left, 0) - COALESCE(cr.ambient, 0) >= 6.0
)
SELECT * FROM classified;
```

The Python tool then runs each row through `ml.v6.regime.classify` to
assign a `LatentState` (v5 rows have `regime IS NULL`; the tool
re-classifies offline using snapshot fields), and computes:

```python
prior[s] = trimmed_mean(
    [row.delta for row in rows if row.state == s],
    trim=0.10,                        # drop top/bottom decile
)
# Bounded to ±2.0 — populations cannot move offsets more than 2 steps
prior[s] = max(-2.0, min(+2.0, prior[s]))
```

### 6.3 Expected priors (sanity)

From audit §2.6 (n=47 left-zone overrides, mean delta likely cool-biased
because cold complaints fire at +deltas, hot complaints at −deltas;
this is the actual bimodal corpus we're asking the prior to summarize):

| State                 | Expected sign | Why                                                                  |
|-----------------------|---------------|----------------------------------------------------------------------|
| `STABLE_SLEEP_DEEP`   | ≈ 0 to −0.5   | early-night cool-biased baselines mostly land near the user prefs    |
| `STABLE_SLEEP_REM`    | ≈ +0.3 to +1  | "cold mid-night" reports (audit §2.2) — user wants warmer            |
| `RESTLESS`            | ≈ −0.5        | restlessness correlates with too-warm (movement_kproxy_right=1.0)    |
| `WAKE_TRANSITION`     | ≈ −1 to −2    | "hot near wake-up" (audit §2.3) — user wants colder                  |
| `HOT_FLASH`           | ≈ −2          | rare event, large adjustment when it does fire                       |
| (`INITIAL_COOL`, `PRE_SLEEP` not learned — see §1)                                                              |

If the empirical prior comes out with the wrong sign, **that is the
finding** — we publish it and inspect the corpus before deploying.

### 6.4 Regeneration cadence

Run quarterly via `python -m ml.learning.derive_population_prior`.
The prior is **not** updated on every night; it is the slow-moving
backstop. The user's per-(user, state) offsets do the fast-moving
work via §5.

## 7. Stability guards

### 7.1 Hold-out validation (per offset, per night)

Last 3 nights' `learning_events` are **held out** from any update
performed *during* those nights — they form a rolling eval set.

At end of night N, before persisting the offset deltas accumulated
during night N, replay nights N-3..N-1 events through *both* the
old offset table and the new (post-N) offset table using
`tools/v6_eval.py` with the proposed `--policy v6_synth_with_offsets`
adapter. Score is **mean discomfort weight** per `discomfort_label.py`
proxy (override label rate + body_sd_q4 minutes).

If `score(new) > score(old) + 0.02` on held-out nights, **reject the
night's offset deltas wholesale** and write `rejected_reason =
"holdout_worse"` to all events for that night. The 0.02 dead-band
prevents noise rejection.

This is the equivalent of v5.1's counterfactual replay (PROGRESS §5,
"NEW MAE 2.13 vs v5 1.81") applied **as a pre-commit gate on the
learner** rather than as a post-hoc lament.

### 7.2 Oscillation guard (per offset)

Maintain a per-(user, state) ring buffer of the last 3 update signs.
If signs alternate (e.g. `+, −, +` or `−, +, −`):

```python
effective_eta(user, state) *= 0.5          # halve until 3 same-sign in a row
```

When 3 consecutive same-sign updates occur, restore to base η.
Persisted in the JSON store. This catches the v4 "EMA oscillator"
failure mode (audit §1, v4 paragraph) without resorting to a global
freeze.

### 7.3 Hard caps (re-stated; enforced on every write)

- `|offset[u, s]| ≤ 3.0`
- `|offset_global[u]| ≤ 2.0`
- `|effective_offset[u, s] + offset_global[u]| → clamped after sum to ±5.0`
  (additional belt for braces; sum cap means a maximally biased state
  on a maximally biased user still cannot move the dial by a full half-cycle).

### 7.4 Kill switch

A new HA helper `input_boolean.snug_learning_enabled` (default `on`).
When `off`, the learning store still observes events (writes them to
PG and updates internal counters) but `get_effective_offset()` returns
`0.0`. Operationally: the learner is collecting data continuously but
the actuation effect is off until user confirms it's pulling correct
direction on the §7.1 metric.

### 8.2 Read/write cadence

- **Read**: once per AppDaemon `initialize()` (hot path = no I/O at
  tick time; the offsets dict lives in process memory).
- **Write**: on every successful `observe()` that produces a non-zero
  `Δoffset`, plus once at `_end_night`. Atomic via `tempfile.mkstemp`
  + `os.replace` (matches existing `_save_learned` pattern,
  `sleep_controller_v5.py:2376-2388`).

### 8.3 CAS lease for end-of-night write

The end-of-night write happens once per night; concurrent calls (e.g.
restart bouncing) must not double-apply caps. Implement as:

```python
with open(LEARNING_NIGHT_LEASE, "x") as f:    # O_CREAT|O_EXCL
    f.write(f"{night_date}:{pid}:{ts}\n")
    try:
        _persist_end_of_night(...)
    finally:
        os.unlink(LEARNING_NIGHT_LEASE)
```

If lease exists and is older than 5 min, treat as stale and steal it.

### 8.4 PG mirror (read-only convenience)

After every write to JSON, also `INSERT INTO learning_offsets_history
(saved_at, offsets_json) VALUES (...)` — append-only, ≤2 KB per row,
~1 row per event. Lets us replay the offset trajectory by night.

## 10. Migration from current learner

### 10.1 Shadow window (1 week)

`ml/learner.py` and `_learn_from_history` continue to run **as-is** for
7 calendar nights. In parallel, the new learner runs in a mode that:

- Captures `AdjustmentEvent` records to PG `learning_events`.
- Computes credits, would-be deltas, would-be offsets.
- Logs `"learning_shadow=" + json.dumps(...)` to AppDaemon log AND to
  `controller_readings.notes` on the override row.
- Does **not** apply offsets (`get_effective_offset` returns 0.0).

The two systems' outputs are written side by side to a new view:

```sql
CREATE OR REPLACE VIEW v_learner_shadow_compare AS
SELECT
    le.t_adjust,
    le.zone,
    le.user_key,
    le.state_at_adjust,
    le.delta             AS event_delta,
    le.offset_delta_applied,                      -- new learner (would-be)
    fh.cycle_adjustments AS old_learner_cycle_adj  -- from _learn_from_history JSON snapshot
FROM learning_events le
LEFT JOIN learner_legacy_snapshot fh
       ON fh.night_date = le.night_date
      AND fh.zone = le.zone
ORDER BY le.t_adjust DESC;
```

### 10.2 Comparison metric (the `tools/v6_eval.py` adapter)

Add `--policy v6_synth_with_offsets` to `tools/v6_eval.py` (per audit
§4 "evaluation framework") that wraps `compute_v6_plan` with the
new learning store. Run after each shadow night:

```bash
python tools/v6_eval.py --policy v6_synth                           # baseline (no learning)
python tools/v6_eval.py --policy v6_synth_with_offsets              # with new learner
python tools/v6_eval.py --policy v5_2_actual                        # current shipped behavior
```

Headline metric: **MAE on held-out nights and override-rate proxy
from `discomfort_label.py`**. Pass criterion:

- New learner MAE ≤ baseline v6 MAE + 0.05
- New learner does **not** propose offsets that would have triggered
  ≥1 additional spurious override (counterfactual override is when the
  proposed setting differs from the user's known final setting by ≥3
  steps in the wrong direction).

## 12. How this fixes audit finding #3

Finding #3 cited three concrete failures of the current learner:

1. *"`_learn_from_history` takes per-(night, cycle) override delta as
   ground truth."* → **Fixed** by §3 credit kernel. The override is
   blame-distributed across the 30-min trail; no single point is
   "ground truth." `(night, cycle)` as a key is also gone — the key
   is `(user, latent_state)`, audit §3.1 trap retired.
2. *"defaults to {} 90% of the time."* → **Fixed** by §6 population
   prior. With zero per-user events the effective offset is the
   population mean, not zero. Day-1 the system already knows
   "WAKE_TRANSITION wants colder."
3. *"30 nights of learning has done nothing."* → **Fixed** by
   `MIN_EVENTS_TO_UPDATE=1` (§5.2) plus the ETA floor that does not
   require ≥150 events. With the population prior, even 0 events
   produces a non-trivial offset. Each event moves the offset by a
   bounded, inspectable amount (§3.3 example: −0.143 + −0.057 from a
   single override).
