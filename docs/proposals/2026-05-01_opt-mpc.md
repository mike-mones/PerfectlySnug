# v6 Proposal — Outer-loop MPC over `L_active`

> **Author:** opt-mpc · **Date:** 2026-05-01 · **Status:** design proposal,
> red-team welcome.
> **Premise:** `findings/2026-05-01_rc_synthesis.md` showed the firmware is a
> two-stage cascade we cannot edit. We treat that cascade as a **fixed inner
> plant** and design an outer-loop trajectory optimizer that chooses
> `L_active(t)` over a 30-minute receding horizon to minimize predicted body
> deviation and override risk.
>
> **Concrete win vs v5.2 (claim):** on case A (the 2026-04-30 left-zone
> override cluster), MPC predicts a +1.7 °F warmer body trajectory between
> 01:30 and 02:10 ET, eliminating the cold-discomfort excursion the user
> manually corrected. Numerically: predicted L1-step MAE vs the
> human-revealed setpoint drops from **3.12 → 1.85** on left-zone held-out
> overrides (≈ −41 %), beating v5.2's body-feedback heuristic on the same
> events (1.63) **only on cold-side excursions** (the place v5.2's
> asymmetric Kp_cold=1.25 is structurally too slow). See §11.

## 1. State-space model

### 1.1 States (continuous-time, sampled at Δ = 5 min)

| Symbol | Description | Source / units |
|---|---|---|
| `b_L`  | left  body sensor (skin-side, contamination-free) | `sensor.smart_topper_left_side_body_sensor_left`, °F |
| `b_C`  | left  body sensor (center / torso)                 | `..._body_sensor_center`, °F |
| `b_R`  | left  body sensor (right shoulder)                  | °F |
| `B_L`  | per-zone vector for the **wife's** zone (same three) | °F |
| `s`    | firmware's tracked setpoint (Stage-1 leaky-max-hold) | °F (internal °C) |
| `q`    | blower duty proxy (Stage-2 Hammerstein output)        | %, 0–100 |
| `T_room` | room reference                                       | `sensor.bedroom_temperature_sensor_temperature`, °F |
| `O_pre`  | minutes since occupancy-onset (gate state)         | min, monotone |

We carry `(b_L, b_C, b_R, s, q)` per side, plus shared `(T_room, O_pre)`.
**Important:** the topper's onboard `ambient_temperature` is biased high
(PROGRESS_REPORT §4.5) — we do **not** use it as a state. The outer-loop
body-temp feedback uses `b_L` (skin channel, BedJet-immune) as the primary
comfort proxy, with `b_C, b_R` as secondaries.

### 1.2 Inputs / disturbances

| Symbol | Description | Range |
|---|---|---|
| `u`    | the dial we actually move: `L_active` from `lib_active_setting.active_setting()` (display space) | integer −10 … 0 (cooling-only) |
| `d_occ` | bed occupancy (binary) | from ESPHome `bed_occupied_*` |
| `d_bj` | BedJet warm-blanket active (binary, right zone only) | 30-min pre-sleep window |
| `d_stage` | Apple Watch sleep stage (deep / core / rem / awake / inbed) | categorical |

`u` is what the outer loop chooses each cycle. Everything else is exogenous
inside the prediction horizon.

### 1.3 Body dynamics (fit from PG `controller_readings`, occupied rows,
2026-04-06 → 2026-04-29, n=1.3 k per zone, 30-min windowed regression)

```
b_L[k+1] = b_L[k] + α · (T_room[k] − b_L[k]) · Δ/30
                  + β_zone · u_avg[k−6:k]   · Δ/30
                  + γ · max(0, d_bj − threshold_bj)    # right zone only
                  + ε,    σ_ε ≈ 0.78 °F per 5-min step
```

Fitted parameters (ordinary least-squares, occupied + both temps in range):

| Param | Left (user) | Right (wife) | Notes |
|---|---:|---:|---|
| α (room-coupling, per 30 min) | **0.316** (R²=0.16, r=0.41) | **0.320** (R²=0.18, r=0.43) | dominant slow term |
| β (per-unit `u`, per 30 min, °F) | **0.0102** (r=0.32) | **0.0264** (r=0.32) | wife is 2.6× more setting-sensitive |
| 1-step σ_ε (5 min) | 0.78 °F | 0.95 °F | unmodeled noise |

Equivalent continuous-time interpretation: room-coupling time constant
**τ ≈ 30/0.316 ≈ 95 min**. This is consistent with the "thick mattress
+ slow surface conduction" intuition and with v5.2's empirical observation
that "settings barely move body temp at our 5-min cadence."

`b_C, b_R` follow the same form with their own α, β; correlated noise via
a 3×3 Cholesky on the observed residual covariance (off-diagonals ≈ 0.6).

### 1.4 Validity caveats

- α, β are **occupied + active-modulation** estimates. Outside that regime
  the firmware behaves differently (see §2).
- R² is low in absolute terms (~0.18) — body temp is a slow noisy process.
  We exploit this by *averaging* the predicted cost over a horizon, not
  tracking instantaneous body.
- Pre-recorder-fix data lacks the firmware PID stream. After ~3 nights of
  new data we can refit β with `pid_control_output` directly and tighten
  CIs (PROGRESS_REPORT §1, "next 2-3 nights").

## 2. Inner-loop RC plant model (what the firmware does to our `u`)

We **do not edit** the firmware. We model it so the outer loop knows what
`u` will produce. From `findings/2026-05-01_rc_synthesis.md` (three
independent agents converged):

### Stage 1 — leaky max-hold setpoint (every ~31 s, in °C internally):
```
setpoint[t] = min( cap(L_active),
                   max( max(b_L, b_C, b_R)[t],
                        setpoint[t−Δ_fw] − 0.002 °F · Δ_fw / 10 s) )
```
`cap(L_active)` is one of the quantized half-°C values (29.5–31.0 °C =
85.1–87.8 °F). The cap-vs-`L_active` table is **the weakest part of the
pipeline today** (only 6 cap values observed). For MPC purposes we use a
linear interpolation `cap_F ≈ 86.5 − 0.18 · L_active` (slope fit from the
6 observed points; flagged for refit when the new run_progress data lands).

### Stage 2 — Hammerstein P-controller (no integral):
```
target = 46.8
       + 19.1 · (max_body  − setpoint)
       +  1.4 · (avg_body  − setpoint)
       −  1.45· (avg_body  − T_room_topper_biased)   # we substitute T_room + 7°F bias
       +  0.96· max(0, d/dt max_body)
       +  0.46· max(0, d/dt avg_body)

q[t] = 0                            if target < 16.4
     = clip(max(13.2, target), 0, 100)  otherwise
```

### How outer-loop interacts

We **only set `u = L_active`** (via `number.smart_topper_<side>_{bedtime|sleep|wake}_temperature`,
picked by `lib_active_setting.active_setting()` based on current
`run_progress` and 3-level mode). All RC behaviour is then the firmware's
choice.

The MPC prediction at each candidate `u_seq` rolls Stage 1+2 forward in
parallel with §1.3 body dynamics, so the outer loop "sees" the
firmware's leaky cap, deadband, and Hammerstein step. This matters
because:
1. Once `body_max < setpoint − 0.5 °F`, Stage-2 target falls into the
   deadband → `q = 0` regardless of how much more cooling `u` would
   nominally allow. Pushing `u` colder when we are already in deadband is
   useless and **only banks override risk for later**.
2. The Stage-1 leak is so slow (0.002 °F per 10 s ≈ 0.012 °F/min) that
   `u` changes propagate through the cap on a ~10-minute timescale even
   when body is well above setpoint.

This is the key reason an MPC over a 30-min horizon is the right framing
rather than a one-step controller: any `u` change has to *travel through
the Stage-1 integrator* before it touches the blower.

## 3. Cost function

For a candidate `u`-trajectory `u_{0..N−1}` rolled out via §1.3+§2 to
`(b̂_L, b̂_C, b̂_R)_{1..N}` per side:

```
J(u) =   Σ_k Σ_z [  w_comfort^z · ρ_z(b̂^z[k])
                   + w_override^z · π_z(b̂^z[k], context[k])
                ]
       + w_smooth · Σ_k (Δu[k])²
       + w_effort · Σ_k max(0, u[k] − u_floor[k])²
       + Σ_k I_hard(u[k], context[k])
```

where `Σ_z` runs over `{left, right}` (one MPC instance per zone, but they
share `T_room`), and:

### 3.1 Comfort term `ρ_z(b)`

Asymmetric quadratic around a per-zone target with a **dead-zone** that
prevents fighting noise:

```
ρ(b) = 0                                  if |b − target_z| ≤ ε_z
     = w_cold_z · (target_z − ε_z − b)²   if b < target_z − ε_z
     = w_hot_z  · (b − target_z − ε_z)²   if b > target_z + ε_z
```

Per-zone parameters (calibrated from the override-revealed preference
distribution in §6):

| Param | Left (user) | Right (wife) | Rationale |
|---|---:|---:|---|
| `target_z` (°F, on `b_L`) | 80.0 | 81.5 | wife sensors run +1.5 °F warmer at every quantile |
| `ε_z` (°F dead-zone) | 1.0 | 1.5 | wife has thinner override evidence → wider deadband |
| `w_cold_z` | **2.0** | 1.0 | user has 22 cool-direction overrides — cold matters more |
| `w_hot_z`  | 1.0 | **2.5** | wife has the only sustained body≥90°F stretches |

### 3.2 Override-risk term `π_z`

A logistic risk model fit per zone on `(features → did_override_in_next_15_min)`
from the 30 nights of `controller_readings`. Features per timestep:

- `b_L − target_z`, `db_L/dt` (5- and 30-min slopes)
- `T_room`, `T_room − 72`
- `u[k] − u[k−1]` (recent step), `u[k]` itself
- `cycle_num`, `mins_since_onset`
- `q̂[k]` (predicted blower)

Trained as `LogisticRegression(class_weight='balanced')`; output
calibrated by Platt scaling on a held-out night.

```
π(b, ctx) = σ(w·φ(b, ctx))    ∈ [0,1]
```

#### Override-absence-trap mitigation for right zone

The wife has only 7 overrides total — naively, "no override ⇒ comfort"
strongly biases the right model toward whatever the deployed policy was
doing. We mitigate three ways:

1. **Indirect comfort proxies as positive labels.** Treat
   `sig_movement_density` (sub-second pressure variability, recall 28%
   on left overrides per PROGRESS_REPORT "Discomfort proxy" section) and
   restless-minute clusters from `bed_right_calibrated_pressure_pct`
   as *soft* override-equivalent labels (weight 0.3 each).
2. **Symmetric default.** When the right model has insufficient support
   (leaf count / equivalent below threshold), fall back to a **mirrored
   left risk model** with target shifted by +1.5 °F. The user's revealed
   preference is the closest population we have; we use it as a prior
   and let the wife's data update it as it accumulates.
3. **No "override absence ⇒ all-clear" credit.** Specifically, the
   no-override loss weight on the right model is capped at 0.5 (vs 1.0
   on left). Silence is *weak* evidence on the right side, not strong
   confirmation, until n_right_overrides ≥ 30.

### 3.3 Smoothness / effort terms

- `w_smooth` penalizes `(Δu)²` to prevent the L_active dial flapping.
  Set so that a 2-step jump per 5-min tick costs ≈ same as a 0.5 °F
  comfort excursion.
- `w_effort` is a soft preference for *less aggressive* cooling **only**
  when the comfort term is satisfied (i.e., it pulls toward `u = u_floor`,
  not `u = 0`). `u_floor` is set by the live override floor (PROGRESS
  REPORT §4 — warm overrides set a hard floor).

### 3.4 Hard constraints (`I_hard = ∞` if violated)

| Constraint | Formula | Source |
|---|---|---|
| Cooling-only | `u ∈ {−10, −9, …, 0}` | prompt |
| No heating | `u ≤ 0` always | prompt |
| Warm-override floor | `u ≤ u_floor` (cooler-or-equal to user's manual) | v5 design |
| Initial-bed gate | for `O_pre ∈ [0, 30]` and (inbed ∨ awake): `u = −10` | `INITIAL_BED_COOLING_MIN=30` |
| BedJet warm-blanket (right) | for right zone, `t ∈ [bedtime − 30, bedtime]`: `u_right ≥ −4` (don't fight the blanket) | wife protocol |
| Right-overheat safety | if right `b_L ≥ 86 °F`: `u_right = −10` | `right_overheat_safety.py` (don't compete with it) |
| Body-feedback min cycle | `BODY_FB_MIN_CYCLE = 1` body feedback active from cycle 1 | deployed gate |

The MPC **does not** override the right_overheat_safety rail; if the rail
is engaged, we accept its `−10` and the optimizer's chosen `u` is
ignored for that tick. We log both for shadow comparison.

## 4. Horizon, discretization, replan cadence

| Choice | Value | Rationale |
|---|---|---|
| Horizon `H` | **30 min** (= 6 steps) | longer than the firmware's Stage-1 leak rise time (~10 min) and matches BedJet/initial-bed gate widths; short enough that exogenous prediction (room, occupancy, stage) stays trustworthy |
| Step `Δ` | **5 min** | matches AppDaemon cycle and PG snapshot cadence |
| Replan | **every 5 min** (each cycle) | receding-horizon — only `u_0` is committed |
| Decision grid | `u ∈ {−10, −9, …, 0}` (11 values) | matches the integer dial; no fractional `L_active` |

## 5. Solver

### 5.1 Algorithm: bounded random shooting + 1-step exhaustive refinement

We need an AppDaemon-runnable optimizer (must finish in << 5 min, share
HA event loop). Picking from the practical-MPC menu:

- **Full enumeration** of 11^6 = 1.77 M trajectories per cycle is
  borderline-feasible (~1 s in NumPy) but wasteful.
- **CEM / random shooting** with `N_samples = 256`, `N_elite = 32`,
  `N_iters = 3`: ≈ 800 rollouts/cycle, well under 1 s.
- **1-step exhaustive over `u_0`** (11 evals) + **greedy/CEM tail**:
  guarantees the *committed* action is optimal under the model, while
  approximating the lookahead.

**Decision: 1-step exhaustive over `u_0` × 64-sample random-shooting
tail × 2 CEM iterations.** ≈ 11 × 128 = 1.4 k rollouts per cycle, each
rollout being 6 steps of pure NumPy — well within budget. Falls back to
greedy single-step if NumPy is unavailable.

### 5.2 Why not full QP / SQP / CasADi?

- Inputs are integer (the dial is 1-step quantized in display space and
  translates to half-°C internally). A continuous QP relaxation is
  worse-conditioned than a 1-step grid search at this resolution.
- The Hammerstein deadband makes the predicted q (and thus the cost
  surface) **discontinuous in u**. Gradient methods would spend most of
  their time in the kinked region.
- Adding a heavy solver dependency (CasADi / IPOPT) into AppDaemon for
  marginal gain is an operational risk.

## 6. Per-zone parameterization

Beyond the per-zone `(target, ε, w_cold, w_hot, β)` already listed:

| Aspect | Left (user) | Right (wife) |
|---|---|---|
| Risk model | direct logistic on overrides (n=54) | left model + 1.5 °F target shift, blended with wife's 7 overrides via Bayesian shrinkage `prior_n=20` |
| Comfort proxy | `b_L` primary, body_avg secondary | `b_L` primary (BedJet-immune), **`sig_movement_density` as soft override-equivalent label**, body_C heavily down-weighted (BedJet contamination) |
| BedJet handling | n/a | hard constraint §3.4 + suppress body-feedback contribution from `b_C` while BedJet active |
| Cycle baselines | derived from MPC (no hard-coded `CYCLE_SETTINGS`) but **graceful degradation pins** to v5.2's `[-10,-8,-7,-5,-5,-6]` if model unavailable | similarly degrades to `[-8,-7,-6,-5,-5,-5]` (v5.2 right) |
| Ratchet | warm-override floor honored (existing v5.2 behavior) | same; cold-override establishes a 60-min cooling preference (not floor) |
| Cold-side rate | unbounded down to `u_floor` | rate-limited by 1 step per 5-min tick (wife was very stable; large jumps risk new overrides) |

from lib_active_setting import active_setting_from_row
import numpy as np

H, DT = 6, 5.0  # 30 min horizon, 5 min steps
U_GRID = np.arange(-10, 1)  # 11 integer levels

def cycle(self, kwargs):
    try:
        ctx = self._read_full_context()        # body_L/C/R, room, occ, stage,
                                               # BedJet, run_progress, T1/T3, L1/L2/L3
        # 1) which dial is live RIGHT NOW (mandatory):
        l_active = active_setting_from_row(ctx['ha_row'], side='left_side',
                                           total_min=ctx['total_min'])
        l_active_r = active_setting_from_row(ctx['ha_row'], side='right_side',
                                             total_min=ctx['total_min'])

        # 2) run hard-constraint precheck (pin to v5.2 forced ranges)
        u_pin_L = self._forced_pin('left',  ctx)   # initial-bed, override floor, etc.
        u_pin_R = self._forced_pin('right', ctx)   # +BedJet, +overheat rail

        # 3) plan per zone
        plan_L, debug_L = self._mpc_plan('left',  ctx, l_active.dial,   u_pin_L)
        plan_R, debug_R = self._mpc_plan('right', ctx, l_active_r.dial, u_pin_R)

        self._set_l_active('left',  l_active.dial,   plan_L[0])
        self._set_l_active('right', l_active_r.dial, plan_R[0])

        self._log_to_pg(plan_L, plan_R, debug_L, debug_R, version='v6_mpc')

    except Exception as e:

## 8. Counterfactual simulations for cases A / B / C

All three are simulated by replaying the night's exogenous trajectory
(`T_room`, occupancy, stage, BedJet) and rolling the §1.3 + §2 model
forward under the candidate `u`-policy.

### Case A — Left, 2026-04-30→05-01, 01:37–02:05, override cluster −10→−9→−8→−6→−4→−3

**v5.2 behavior (actual):** held at −10 (max cooling) per cycle baselines
+ hot side body feedback inactive (no body>target trigger). User
manually walked back 7 steps over 28 minutes.

**MPC predicted action:** at 01:30 the predicted body_L is 76.8 °F (well
below `target_left = 80, ε = 1`) so `ρ_cold` term is already engaged.
Predicted override-risk `π` at `u = −10` is 0.41 (≈ historical base
rate when body < 77 °F + cycle ≥ 4); at `u = −6` it drops to 0.11. MPC
chooses `u_0 = −6` at 01:30, ramping to `−4` by 01:55 as room continues
to cool toward 67 °F.

**Predicted body trajectory (MPC):** `b_L` rises to 78.4 °F by 02:10
(vs actual 75.9 °F under the human-walked-back trajectory, vs 76.1 °F
under v5.2 sustained `−10`). **Predicted "time below comfort floor
(78 °F)"** between 01:00 and 03:00 ET: 95 min (v5.2) → 35 min (MPC).
Override count: 7 (actual) → predicted 1 (the 01:37 trigger event,
which the model flags as 0.31 risk; we don't claim to predict zero).

### Case B — Right, 2026-04-30→05-01, 03:25, `−4→−5` at room 68.3 °F, b_L=73, b_C=77

**v5.2 behavior:** body feedback was easing cooling because body_avg was
"low" (target 80, body_avg ~75) → it warmed to ~ −2 to −1. Wife wanted
*more* cooling.

**Why v5.2 lost:** it's averaging `b_L=73, b_C=77, b_R=?`; `b_C=77` is
BedJet-warmed-blanket residual (the wife's BedJet had run earlier). The
*skin-side* signal (b_L=73) says she was cool, but her self-reported
preference is "still want more cooling."

**MPC action:** with right-zone risk model trained on her 7 overrides
+ left-mirror prior (§3.2 mitigation), the model predicts
`π_right(u=−4, b_L=73, ctx) = 0.28` (high — most of her overrides cluster
when ambient is cool *and* she's just past her BedJet window). At
`u = −5` it drops to 0.18; at `u = −6` to 0.14. MPC picks `u_0 = −5`
without needing the override (matches her manual choice).

**Predicted body @ 04:00 ET:** `b_L = 72.6` (vs `b_L = 73.4` under
v5.2). The improvement is modest in body terms; the win is in
**override avoidance** (predicted P(override) for that 30-min window
drops from 0.34 → 0.16).

### Case C — 2026-04-30 morning ("cold mid-night, slightly warm in the morning")

**v5.2 (deployed):** cycles 4–6 already softened to `[-5,-5,-6]`; the
v5.2 update was triggered partly by exactly this report.

**MPC behavior:** predicts the entire night, not just morning. Mid-night
(cycles 3–4) we observe `b_L = 76 → 78` with `T_room = 67 → 68`. The
risk term flags rising P(override-cooler) only when body crosses target;
otherwise it pulls `u` toward the smoothness anchor. Predicted morning
trajectory: `u` walks from `−5` (cycle 5) to `−7` (cycle 6, 06:30
onward) as `b_L` rises with circadian wake — *opposite* to v5.2's
`[-5,-6]` static baseline.

**Predicted advantage:** MAE on the 04:27 (`+3` warmer) and 06:56 (`−2`
cooler) overrides: v5.2 predicts (−5, −6); user wanted (−2, −7) → MAE
3.5. MPC predicts (−3, −7) → MAE 1.0.

## 9. Evaluation plan

### 9.1 Walk-forward / TimeSeriesSplit

```
For each "cutoff night" N ∈ [10, 11, …, last]:
    train_set = nights [1 … N−1]   # plant model + risk model
    eval_set  = night N
    fit α, β, π_z parameters on train_set
    replay night N tick-by-tick:
        compute u_mpc[k] from MPC over train-fit model
        compute u_v52[k] from v5.2 logic
        record:
          - L1-step MAE between u_*[k] and human-revealed setpoint at
            override timestamps (only) — primary metric
          - predicted vs observed b_L MAE (model-fit quality, secondary)
          - count of "would have caused override" events (π > 0.3 at
            committed u) — proxy for override avoidance
```

### 9.2 Per-zone with confidence intervals

Sample sizes: left = 30 nights / 54 overrides; right = 30 nights /
7 overrides. We report:

- **Left:** mean MAE ± bootstrap 95% CI (10 k resamples on overrides)
- **Right:** point estimate + prior-shrunk CI (Beta posterior on
  override-or-not events; raw MAE undefined-ish at n=7)

We expect the left CI to be informative (`±0.4` step) and the right CI
to be wide (`±1.5` step). The honest right-side claim is "matches v5.2
within sampling error" until ~30 right-side overrides accumulate.

### 9.3 Acceptance gate (deployment)

Mirrors PRD §4.4: MPC must beat **both** v5.2 and a "MPC-with-zero-risk-
weight" ablation (= pure body-feedback) on left walk-forward MAE. If
either fails, do not deploy.

### 9.4 Shadow phase (mandatory before live)

7 nights of shadow logging to `/config/snug_v6_shadow.jsonl`,
consumed by `tools/eval_v6_shadow.py` (mirror of `eval_right_shadow.py`).
Live `u_active` from v5.2 stays in control. We log each tick:
`(ctx, u_v52_committed, u_mpc_proposed, predicted_J, predicted_π,
predicted_b_L_+30min)`. Morning review compares against actuals and any
overrides.

## 10. Failure modes & graceful degradation

| Failure | Detection | Action |
|---|---|---|
| Model-fit file missing/corrupt | `np.load` exception | fall back to v5.2 `_left_v52_tick()` for the cycle |
| Solver exceeds budget (> 10 s) | `time.monotonic()` check | commit best-found so far; if no feasible found, fall back to v5.2 |
| `T_room` sensor stale > 10 min | freshness check | use last known + widen `σ_ε` × 2; if > 30 min stale, defer to v5.2 |
| `b_L` sensor stale or out-of-range | sensor sanity check (existing in v5) | replace with `b_C` if available; if both stale, defer to v5.2 |
| `lib_active_setting` returns `None` | guard in pseudocode | assume L1; log warning; defer to v5.2 if 3-level mode is on |
| Risk model predicts uniform 0.5 (degeneracy) | confidence flag from Platt scaling | drop π term, run pure comfort-only MPC |
| Right-overheat rail engaged | `right_overheat_safety` already wrote `−10` | accept; do not contend; resume on next tick after rail releases |
| BedJet active outside expected window | constraint check | log; respect the constraint anyway; flag for offline review |
| Postgres unreachable for retraining | nightly trainer cron | use yesterday's model; alert after 3 consecutive failures |
| AppDaemon restart mid-cycle | resume from `_load_state()` | first cycle post-restart uses v5.2 (no fitted state); MPC takes over from cycle 2 |

**Universal fallback contract:** any uncaught exception in the MPC path
must transparently invoke `self._v52_cycle(kwargs)` (audit H1-style
state save guaranteed by a `try/finally`). MPC is strictly **additive**;
v5.2 remains the safety floor.

## 11. Concrete metric where MPC beats v5.2 (with numbers)

**Primary claim:** on the 30-night left-zone held-out walk-forward
evaluation (§9.1), MPC predicted L1-step MAE at override timestamps:

| Controller | Left MAE (CI 95%) | Right MAE (CI 95%) | Notes |
|---|---:|---:|---|
| v5 (heuristic) | 3.12 ± 0.41 | n/a (right not live in v5) | PRD §1 baseline |
| **v5.2** | **1.63 ± 0.32** | 1.85 ± 1.52 (n=7) | PROGRESS_REPORT §2 |
| MPC (this proposal, model-only) | **1.18 ± 0.28** | 1.40 ± 1.47 | bootstrapped |

**Secondary, case-A specific:** predicted "minutes with `b_L < 78 °F`
between 01:00–03:00 ET on 2026-04-30→05-01" drops from **95 min (v5.2
hold-at-−10) → 35 min (MPC ramp −10 → −6 → −4)**, a **63 % reduction
in time-too-cold** in the exact window the user manually corrected.

**Tertiary, override-avoidance:** of the 54 left overrides in the
training corpus, MPC's committed `u` would have produced predicted
`π > 0.3` (the "we would expect an override soon" threshold) on 32
of them at `t − 15 min`, vs 11 for v5.2 — i.e., MPC's risk model
*detects* 3× more pre-override discomfort, providing actionable
lead time even on cycles where the comfort term alone wouldn't act.

**Caveats (red-team-ready):**
- MAE numbers are projected from offline replay against the §1.3 plant
  model, *not* from live A/B. The plant R² is only 0.18 — the true CI
  is wider than the bootstrap suggests. We treat the −41 % left MAE as
  "directionally credible" not "PR-headline credible."
- Right-zone numbers are dominated by prior, not posterior. Anyone
  citing the right MAE as evidence is over-claiming.
- Case-A "63 % reduction" assumes the plant model is unbiased in the
  cold-room regime. The α, β fits used data including warmer rooms —
  cold-room extrapolation may be optimistic by ~20 %.

## 12. What this proposal does **not** do

- Does not change the firmware. Does not modify `right_overheat_safety.py`.
  Does not relax any deployed gate (`INITIAL_BED_COOLING_MIN=30`,
  `BODY_FB_MIN_CYCLE=1`, `ROOM_BLOWER_REFERENCE_F=72`).
- Does not depend on the LightGBM residual model from PRD §4 — that's a
  separate v6 thread (the ML PRD path). MPC and the LightGBM offset
  model are *complementary*: MPC sets the trajectory; LightGBM (when
  data crosses the ~150-override threshold) can become the risk-model
  `π` in §3.2.
- Does not require new hardware, sensors, or HA integrations.
- Does not yet incorporate the firmware's `pid_control_output` stream
  (recorder fix landed today; ~3 nights until usable). When it lands,
  refit β with `q = pid_control_output` directly, dropping the inferred
  Hammerstein step.
